# BFF (Backend for Frontend)

## Краткий ответ для собеседования

**BFF** — слой бэкенда, который делают **специально под конкретный клиент** (веб-приложение, мобильное приложение, умный телевизор). Он принимает запросы от этого клиента, обращается к общим сервисам/микросервисам или к агрегатору, **собирает и преобразует данные** под нужды экрана и отдаёт один ответ в удобном для фронта формате. Зачем: **меньше запросов** с клиента (один запрос вместо десятка к разным API), **формат под клиент** (не «универсальный» API, а именно то, что нужно странице), **скрытие сложности** бэкенда (фронт не знает про множество микросервисов). Для фронтенд-разработчика: вместо вызова нескольких API и склейки на клиенте — один запрос к BFF, который уже возвращает готовые данные для экрана. На собеседовании: что такое BFF, зачем он, чем отличается от «одного общего API», когда уместен.

---

## Что такое BFF простыми словами

**Backend for Frontend** — это **отдельный серверный слой**, который стоит между фронтендом (или другим клиентом) и остальным бэкендом. Он не содержит основную бизнес-логику и базы данных — он **адаптирует** то, что отдают другие сервисы, под конкретный клиент.

Идея: у тебя есть веб-сайт, мобильное приложение и, например, приложение для ТВ. Им нужны **разные данные и разный формат**: вебу — много полей для сложной страницы, мобилке — компактный ответ и меньше данных, ТВ — другой набор. Вместо того чтобы один «общий» API пытался угодить всем или фронт дергал пять разных микросервисов и сам склеивал результат, делают **отдельный BFF под каждый тип клиента** (или один BFF с разными эндпоинтами под веб/мобилку). BFF ходит к нужным сервисам, собирает данные, режет лишнее, склеивает в один ответ и отдаёт клиенту уже «готовую порцию» под экран.

Простыми словами: **BFF — переводчик и сборщик**: фронт говорит «дай данные для страницы профиля», BFF идёт в сервис пользователей, в сервис заказов, при необходимости в кэш, собирает один JSON под эту страницу и отдаёт. Фронт делает один запрос и рисует экран.

## Зачем нужен BFF

- **Меньше запросов с клиента** — вместо 5–10 запросов к разным API (пользователь, заказы, рекомендации, уведомления…) фронт делает **один** запрос к BFF. BFF сам дергает нужные сервисы (параллельно или по цепочке), собирает ответ и отдаёт. Быстрее загрузка, проще код на клиенте, меньше проблем с водой (waterfall) запросов.
- **Формат под клиент** — ответ BFF можно спроектировать **под экран**: те поля и та вложенность, которые нужны странице, без лишнего. Не нужно подстраивать «общий» API под десяток клиентов — у каждого клиента (или типа клиента) свой BFF или свои эндпоинты.
- **Скрытие внутренней архитектуры** — за BFF могут стоять микросервисы, графы, legacy-системы. Фронт не знает, откуда данные пришли и сколько сервисов задействовано — он знает только контракт BFF. Переименовали сервис, разбили на два — меняется только BFF, клиенты не трогаем.
- **Разные BFF под разные клиенты** — веб-BFF может отдавать большие ответы и кэшировать по-своему, мобильный BFF — сжимать и упрощать. Один бэкенд «для всех» часто превращается в компромисс; BFF даёт возможность оптимизировать под конкретный клиент.

Для фронтенд-разработчика это значит: **один запрос на экран (или на блок экрана)**, один тип ответа, меньше логики склейки и обработки на клиенте. Часто BFF делают те же команды, что и фронт, чтобы контракт между «экраном» и «ответом API» был удобным.

## BFF vs «один общий API»

| Аспект | Один общий API для всех клиентов | BFF под клиента |
|--------|----------------------------------|------------------|
| **Кто подстраивается** | Клиент подстраивается под общий контракт; часто лишние поля или несколько запросов | BFF подстраивает ответ под экран/клиент; один ответ под страницу |
| **Количество запросов с клиента** | Часто много (несколько эндпоинтов под один экран) | Обычно меньше (один-два запроса к BFF на экран) |
| **Изменение бэкенда** | Меняем общий API — могут пострадать все клиенты | Меняем внутренние сервисы — контракт BFF с клиентом можно сохранить |
| **Кто владеет контрактом** | Бэкенд-команда; фронт подстраивается | Часто фронт/полный стек команда; BFF как «свой» слой под свой клиент |

BFF не заменяет бизнес-логику и микросервисы — он **оркестрирует** вызовы к ним и **адаптирует** результат под фронт.

## Когда уместен BFF

- **Много микросервисов** — фронту неудобно вызывать каждый сам; BFF агрегирует.
- **Разные клиенты** (веб, мобилка, партнёры) — разным клиентам разный формат и объём данных; отдельный BFF (или слой в BFF) под каждый тип.
- **Сложные экраны** — одна страница требует данных из нескольких доменов; один запрос к BFF вместо графа запросов с клиента.
- **Команда владеет и фронтом, и «своим» бэкендом** — BFF часто пишут и поддерживают фронтенд- или полный стек команды, контракт заточен под их экраны.

Когда BFF может быть избыточен: простое приложение, один монолитный бэкенд, один тип клиента и мало запросов на экран — тогда один API может быть достаточен.

---

## Как это связано с фронтендом

- Фронт обращается **только к BFF** (или в основном к BFF) для данных экрана. Не дергает напрямую десять микросервисов.
- **Контракт BFF** можно проектировать «от экрана»: какие данные нужны для страницы профиля, корзины, каталога — такой ответ и возвращает BFF. Удобно для типизации (TypeScript) и для работы с TanStack Query / SWR (один ключ запроса — один ответ под экран).
- **Ошибки и загрузки** — один запрос: один loading, одна обработка ошибки. Не нужно склеивать состояния по нескольким запросам.
- **Аутентификация** — BFF может подставлять токен/сессию при вызовах к внутренним сервисам; клиент шлёт куки или один токен в BFF.

На собеседовании можно сказать: «BFF — слой между фронтом и бэкендом, который собирает данные из разных сервисов и отдаёт один ответ под экран. Фронт делает меньше запросов, получает формат под себя, не зависит от внутренней архитектуры бэкенда.»

## Кратко запомнить

- **BFF** — Backend for Frontend: слой бэкенда **под конкретный клиент**, агрегирует вызовы к сервисам и отдаёт **один ответ под экран**.
- **Зачем:** меньше запросов с клиента, формат под клиент, скрытие внутренней архитектуры, разный BFF под веб/мобилку.
- **Для фронта:** один (или мало) запросов на экран, готовые данные под страницу, проще типизация и обработка загрузки/ошибок.
- **Не подменяет** бизнес-логику и микросервисы — оркестрирует их и адаптирует ответ под фронт.
