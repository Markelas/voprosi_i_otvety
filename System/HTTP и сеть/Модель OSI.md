# Модель OSI (Open Systems Interconnection)

## Краткий ответ для собеседования

**Модель OSI** — это эталонная семиуровневая модель сетевого взаимодействия. Каждый уровень решает свою задачу: от физической передачи сигнала до протоколов приложений (HTTP, DNS). На практике чаще используют **модель TCP/IP** (4 уровня), но вопросы про «слои» и «на каком уровне что» обычно задают в контексте OSI. Фронтенд-разработчик в основном работает с **уровнем 7 (приложение)** — HTTP, API, WebSocket; полезно понимать, что под ним — TLS (6–5), TCP (4), IP (3).

---

## Зачем это на собеседовании

- Показать понимание «стека»: запрос браузера проходит через несколько уровней (HTTP → TLS → TCP → IP → среда).
- Объяснить, где что живёт: HTTP — приложение, TLS — шифрование/представление, TCP — надёжная доставка, IP — адресация и маршрутизация.
- Связать с реальностью: CORS, HTTPS, сокеты, «почему один медленный запрос блокирует другие» (TCP/HTTP) — всё укладывается в слои.

---

## Семь уровней OSI (сверху вниз)

| № | Уровень        | Назначение                     | Примеры для фронтенда / интернета   |
|---|----------------|--------------------------------|-------------------------------------|
| 7 | **Приложения** | Протоколы приложений, API      | HTTP, HTTPS, WebSocket, DNS, REST   |
| 6 | **Представления** | Формат данных, шифрование   | TLS/SSL, кодирование (JSON, gzip)   |
| 5 | **Сеансовый**  | Сеанс, диалог между приложениями | Сессии, RPC, часть логики TLS      |
| 4 | **Транспортный** | Доставка данных между узлами | TCP, UDP, порты                    |
| 3 | **Сетевой**    | Маршрутизация, адресация       | IP, роутеры                         |
| 2 | **Канальный**  | Передача кадров в пределах сети | Ethernet, MAC-адреса, коммутаторы  |
| 1 | **Физический** | Сигнал, среда передачи         | Кабель, Wi-Fi, радиосигнал          |

Запоминание (сверху вниз): **П**риложение — **П**редставление — **С**еанс — **Т**ранспорт — **С**еть — **К**анал — **Ф**изика (ППСТСКФ). Или снизу вверх: «Физик Каналу Сети Транспорт Сеанс Представил Приложение».

## Что важно для фронтенда по уровням

### Уровень 7 — Приложение (Application)

- **HTTP/HTTPS** — запросы к API, загрузка страниц, ресурсов.
- **WebSocket** — постоянное соединение для чатов, уведомлений, реального времени.
- **DNS** — преобразование доменного имени в IP (часто говорят «работает на прикладном уровне» или «поверх UDP»).
- **REST, GraphQL** — способы организации API на уровне приложения.

На собеседовании: «На каком уровне работает HTTP?» — **уровень 7 (приложение)**.

---

### Уровень 6 — Представление (Presentation)

- **Шифрование и расшифровка** — в вебе это в первую очередь **TLS/SSL** (сертификаты, HTTPS).
- Кодирование данных (например, сжатие gzip, форматы) тоже относят сюда.

На собеседовании: «Где в модели OSI «живёт» HTTPS / TLS?» — поверх прикладного протокола, по смыслу — **уровни 6–7** (часто говорят «приложение + представление»).

---

### Уровень 5 — Сеансовый (Session)

- Управление сеансом связи (установка, поддержание, завершение).
- В вебе сессии (session) на сервере, cookie, JWT — это уже логика приложения, но идея «сеанса» из OSI сюда привязана.
- На собеседовании упоминают реже; достаточно знать, что это «слой сеанса».

---

### Уровень 4 — Транспортный (Transport)

- **TCP** — надёжная доставка, порядок пакетов; **HTTP обычно работает поверх TCP**.
- **UDP** — без гарантий, быстрее; используется, например, в DNS, стриминге, играх.
- **Порты** — идентификация приложения на хосте (80 — HTTP, 443 — HTTPS).

На собеседовании: «На каком уровне работают TCP и UDP?» — **уровень 4 (транспортный)**. «Почему один тяжёлый запрос может блокировать другие?» — можно сказать про TCP (head-of-line blocking) и про HTTP/1.1 (очередь запросов).

---

### Уровень 3 — Сетевой (Network)

- **IP** — адресация (IP-адреса), маршрутизация пакетов между сетями.
- Роутеры работают на этом уровне.

Для фронтенда: понимать, что у сервера и клиента есть IP, а доменное имя через DNS превращается в IP на уровне приложения, а дальше пакеты идут по IP.

---

### Уровни 2 и 1 — Канальный и Физический

- **Канальный (2):** передача кадров внутри одной сети (Ethernet, Wi-Fi), MAC-адреса.
- **Физический (1):** среда — кабель, радиоканал, биты.

На собеседовании фронтенда спрашивают редко; достаточно ответить: «Канальный — кадры и локальная сеть, физический — среда передачи».

---

## Модель TCP/IP (4 уровня) — как в реальности

В интернете чаще говорят о **стеке TCP/IP**:

| TCP/IP        | Что входит              | Аналог в OSI    |
|---------------|--------------------------|------------------|
| Приложение    | HTTP, DNS, WebSocket…   | 7, 6, 5          |
| Транспорт     | TCP, UDP                 | 4                |
| Интернет      | IP                       | 3                |
| Канал + среда | Ethernet, Wi-Fi, кабель  | 2, 1             |

«HTTP работает поверх TCP поверх IP» — это и есть движение по стеку вниз (приложение → транспорт → сеть).

---

## Путь запроса от браузера (упрощённо)

1. **Приложение (7):** браузер формирует HTTP-запрос (URL, метод, заголовки).
2. **Представление (6):** при HTTPS добавляется TLS — шифрование.
3. **Транспорт (4):** данные упаковываются в TCP-сегменты (порты, порядок).
4. **Сетевой (3):** TCP упаковывается в IP-пакеты (IP-адреса отправителя и получателя).
5. **Канал/физический (2–1):** пакеты уходят в сеть (Ethernet, Wi-Fi и т.д.).

Ответ идёт в обратном порядке: физический сигнал → кадры → IP → TCP → TLS → HTTP → приложение (браузер).

## Каверзные и типичные вопросы

**На каком уровне OSI работает HTTP?**  
— Уровень 7 (приложение).

**На каком уровне TCP и UDP?**  
— Уровень 4 (транспортный).

**Где в модели OSI TLS/SSL?**  
— По смыслу — уровень представления (6), часто рассматривают вместе с прикладным (7), т.к. TLS оборачивает HTTP.

**Чем модель OSI отличается от TCP/IP?**  
— OSI — 7 уровней, эталонная модель; TCP/IP — 4 уровня, реальный стек интернета. Уровни TCP/IP объединяют несколько уровней OSI (например, приложение в TCP/IP = 7+6+5 в OSI).

**Почему знание уровней полезно фронтенд-разработчику?**  
— Понимание, что запрос к API (HTTP) идёт поверх TLS, TCP, IP; откуда берутся задержки, блокировки (TCP, HTTP/1.1), как связаны CORS, HTTPS, сокеты и т.д.

**Что такое порт и на каком уровне?**  
— Порт идентифицирует приложение на хосте (например, 80 — HTTP, 443 — HTTPS). Уровень 4 (транспортный); порты задаются в TCP/UDP.

---

## Кратко запомнить

- **OSI — 7 уровней**, от приложения (HTTP, DNS) до физики (кабель, сигнал).
- **Фронтенд в основном на 7-м** (HTTP, API); важно понимать **4-й** (TCP/UDP) и **6-й** (TLS для HTTPS).
- **TCP/IP — 4 уровня**, то, на чём реально построен интернет; HTTP поверх TCP поверх IP.
- На собеседовании достаточно назвать уровни, привести примеры (HTTP — 7, TCP — 4, IP — 3) и кратко описать путь запроса по стеку.
