# Метрики производительности (LCP, FID, CLS и др.)

## Краткий ответ для собеседования

Основные метрики загрузки и отзывчивости страницы: **LCP** (Largest Contentful Paint) — когда отрисован самый крупный контент на экране; **FID** (First Input Delay), сейчас заменяется на **INP** — задержка до первого отклика на действие пользователя; **CLS** (Cumulative Layout Shift) — «прыжки» контента, нестабильность макета. Вместе они входят в **Core Web Vitals** — Google использует их для оценки UX и SEO. Дополнительно: **FCP** (First Contentful Paint) — первый контент на экране; **TTI** (Time to Interactive) — когда страница полностью интерактивна; **TBT** (Total Blocking Time) — суммарное время, когда главный поток был заблокирован задачами > 50 ms. На собеседовании: назвать метрики, что каждая измеряет, какие пороги «хорошо/плохо», как примерно улучшать.

---

## Core Web Vitals — три главные метрики

Google выделяет три метрики, которые сильнее всего связаны с восприятием качества страницы пользователем. По ним оценивают UX и учитывают в поисковой выдаче (SEO).

---

### LCP — Largest Contentful Paint (крупнейшая отрисовка контента)

**Что измеряет:** момент, когда **самый большой видимый блок контента** (изображение, видео, текстовый блок) отрисован на экране. То есть не «первый пиксель», а «основной контент страницы уже виден».

**Зачем:** пользователь понимает, что страница «загрузилась» по появлению главного элемента — героя, заголовка, большой картинки. Долгий LCP создаёт ощущение медленной загрузки.

**Пороги (ориентир):** хороший LCP — до **2,5 с**; требует улучшения — 2,5–4 с; плохо — больше 4 с.

**Что ухудшает:** медленный сервер или сеть, тяжёлые изображение/видео в LCP-элементе, блокирующий CSS/JS до отрисовки, долгий серверный рендер. **Как улучшать:** оптимизировать загрузку LCP-ресурса (приоритет, прелоад, CDN), уменьшать блокировки (критичный CSS, отложенный JS), ускорять сервер и TTFB.

---

### FID / INP — отзывчивость на первое действие

**FID (First Input Delay)** — задержка между **первым действием пользователя** (клик, тап, нажатие клавиши) и **моментом, когда браузер смог начать обрабатывать** это действие. То есть «насколько быстро страница отреагировала на первый клик».

**Почему задержка:** главный поток браузера в этот момент может быть занят — парсинг JS, выполнение скриптов, layout. Пока он занят, обработчик клика не запускается. FID как раз измеряет эту задержку.

**INP (Interaction to Next Paint)** — метрика, которая приходит на смену FID: учитывает не только первое взаимодействие, а набор взаимодействий за визит и отражает общую отзывчивость. Идея та же: «как быстро страница реагирует на действия пользователя».

**Пороги (ориентир):** хороший FID/INP — до **100 ms**; плохо — больше 100 ms (пользователь чувствует «лаг»).

**Что ухудшает:** долгие задачи в главном потоке (большой JS, тяжёлые вычисления), много скриптов без разбиения. **Как улучшать:** уменьшать и разбивать JS (code splitting, tree shaking), откладывать некритичный JS (defer, lazy), переносить тяжёлое в Web Workers, разбивать длинные задачи.

---

### CLS — Cumulative Layout Shift (накопленный сдвиг макета)

**Что измеряет:** насколько контент **«прыгает»** во время загрузки и после. Каждый сдвиг видимой области (когда элемент сместился между кадрами) даёт балл; баллы суммируются. Высокий CLS — страница «дёргается», легко случайно нажать не ту кнопку.

**Зачем:** стабильный макет — часть хорошего UX. Реклама, изображения без размеров, шрифты, подгружаемый контент часто сдвигают блоки — CLS это фиксирует.

**Пороги (ориентир):** хороший CLS — до **0,1**; требует улучшения — 0,1–0,25; плохо — больше 0,25.

**Что ухудшает:** изображения/виджеты без width/height (место не зарезервировано), реклама или iframe без резерва места, контент, вставляемый выше уже отрисованного, шрифты с заметным FOIT/FOUT. **Как улучшать:** задавать размеры у img/iframe (или aspect-ratio), резервировать место под динамический контент, подключать шрифты с font-display и по возможности предзагружать.

## Другие важные метрики

| Метрика | Что измеряет | Зачем |
|--------|---------------|--------|
| **FCP** (First Contentful Paint) | Момент, когда на экране появился **первый контент** (текст, изображение, фон). | «Что-то уже видно» — первое ощущение загрузки. |
| **TTI** (Time to Interactive) | Момент, когда страница **полностью интерактивна**: контент отрисован, JS выполнен, главный поток свободен, клики обрабатываются быстро. | «Можно пользоваться», а не только смотреть. |
| **TBT** (Total Blocking Time) | Суммарное время, когда главный поток был **заблокирован задачами дольше 50 ms** (от FCP до TTI). | Чем больше TBT, тем дольше страница не может нормально реагировать на ввод. |
| **TTFB** (Time to First Byte) | Время от запроса страницы до **первого байта ответа** от сервера. | Отражает задержку сервера и сети до начала загрузки. |

FCP и LCP про «увидели», TTI про «можем пользоваться», FID/INP про «как быстро отреагировала», CLS про «насколько стабилен макет», TBT — почему долго не отреагировала.

## Сводка: что измеряем и пороги (Core Web Vitals)

| Метрика | Что это | Хорошо | Плохо |
|--------|---------|--------|--------|
| **LCP** | Крупнейший контент отрисован | ≤ 2,5 с | > 4 с |
| **FID / INP** | Задержка отклика на действие | ≤ 100 ms | > 100 ms |
| **CLS** | Накопленный сдвиг макета | ≤ 0,1 | > 0,25 |

---

## Как измеряют

- **В проде:** Chrome User Experience Report (CrUX) — реальные данные пользователей; Google использует их для Core Web Vitals в Search Console и PageSpeed Insights.
- **В разработке:** Lighthouse (в DevTools или CI), панель Performance в Chrome DevTools, расширения. Можно симулировать медленную сеть и слабое устройство.
- **Программно:** API **PerformanceObserver** — события для LCP, FID, CLS и других метрик; можно отправлять в свою аналитику.

## Кратко запомнить

- **Core Web Vitals:** LCP (крупный контент, ≤ 2,5 с), FID/INP (отклик на действие, ≤ 100 ms), CLS (сдвиги макета, ≤ 0,1).
- **LCP** — «главное на экране отрисовано»; **FID/INP** — «страница быстро реагирует»; **CLS** — «макет не прыгает».
- **FCP** — первый контент; **TTI** — полная интерактивность; **TBT** — время блокировки главного потока.
- Измеряют в Lighthouse, Performance, через PerformanceObserver; в проде — CrUX, Search Console.
