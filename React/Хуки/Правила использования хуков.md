# Правила использования хуков (Rules of Hooks)

## Краткий ответ для собеседования

Правила хуков в React: **1) Вызывать хуки только на верхнем уровне** — не внутри условий, циклов и вложенных функций, чтобы порядок вызовов был одинаков при каждом рендере. **2) Вызывать хуки только в функциональных компонентах или в кастомных хуках** — не в обычных функциях и не в классах. **3) Порядок вызова хуков должен быть одинаковым** при каждом рендере — React сопоставляет хуки по порядку (по индексу в списке внутри Fiber). **4) Кастомные хуки должны начинаться с префикса "use"** — так React (и линтер) понимают, что внутри могут быть хуки, и к ним применяются те же правила. Причина: React хранит состояние и эффекты хуков в **связанном списке** по порядку вызова; если порядок меняется (условие, цикл), React «путает» хуки и получаются баги. На собеседовании: перечислить правила и объяснить, что порядок важен из‑за внутреннего хранения по индексу.

---

## Правило 1: Только на верхнем уровне (не в условиях, циклах, вложенных функциях)

**Правило:** Хуки нужно вызывать **непосредственно в теле** функционального компонента или кастомного хука, на верхнем уровне. Не внутри `if`, `for`, `while`, не внутри вложенной функции (например, внутри `setTimeout` или колбэка).

**Почему:** React при каждом рендере вызывает компонент и **по порядку вызова** сопоставляет каждый вызов хука с записью во внутреннем списке (связанный список хуков в Fiber Node). Индекс в этом списке определяется только порядком: первый вызов хука — индекс 0, второй — 1 и т.д. Если в одном рендере хук вызывается (например, из `if (condition)`), а в другом — нет, порядок следующих хуков сдвигается. Тогда второй хук начнёт читать данные первого, третий — второго — состояние и эффекты «перепутаются», возможны краши и неверное поведение.

**Примеры:**

- ❌ Условие: `if (flag) { const [x, setX] = useState(0); }` — при смене `flag` порядок хуков меняется.
- ❌ Цикл: `items.map(() => { const [v, setV] = useState(0); ... })` — в каждом рендере количество вызовов может быть разным.
- ❌ Вложенная функция: `useEffect(() => { const [x, setX] = useState(0); }, []);` — хук вызывается не в теле компонента.
- ✅ Всё на верхнем уровне: все вызовы `useState`, `useEffect` и т.д. в начале функции, без условий и циклов. Условие можно применять к **использованию** значения хука, а не к самому вызову: `const [x, setX] = useState(0); if (!flag) return null;` — ок.

---

## Правило 2: Только в функциональных компонентах и кастомных хуках

**Правило:** Хуки можно вызывать только из **функции, которая является React-компонентом** (возвращает JSX и вызывается React при рендере), или из **кастомного хука** — функции, имя которой начинается с `use` и которая сама вызывает хуки.

**Почему:** React при рендере знает, **какой компонент сейчас выполняется**, и привязывает список хуков к этому компоненту (к его Fiber Node). Обычная функция (вспомогательная, обработчик, утилита) не вызывается React в контексте «сейчас рендерится компонент X» — у неё нет своего Fiber и списка хуков. Классовые компоненты не вызывают хуки как функции при рендере — у них другой механизм (экземпляр класса, методы жизненного цикла). Поэтому хуки работают только там, где React однозначно связывает вызов с конкретным компонентом или кастомным хуком (который в свою очередь вызывается только из компонента или другого хука).

**Примеры:**

- ❌ Обычная функция: `function helper() { const [x, setX] = useState(0); return x; }` — helper не компонент и не хук.
- ❌ Классовый компонент: внутри метода `render()` или другого метода класса вызывать хуки нельзя.
- ✅ Функциональный компонент: `function MyComponent() { const [x, setX] = useState(0); return <div>{x}</div>; }`.
- ✅ Кастомный хук: `function useMyHook() { const [x, setX] = useState(0); return x; }` — вызывается только из компонента или другого хука.

## Правило 3: Одинаковый порядок вызова при каждом рендере

**Правило:** Количество и порядок вызовов хуков при каждом рендере компонента должны быть **одинаковыми**. Первый вызов — всегда один и тот же хук, второй — всегда следующий и т.д.

**Почему:** React не хранит состояние «по имени» хука или переменной — он хранит **список хуков по порядку** в структуре Fiber (связанный список в `memoizedState`). При вызове `useState(0)` React смотрит: «сейчас по счёту первый вызов хука в этом компоненте — беру первую ячейку списка». При следующем вызове — вторую ячейку. Если в одном рендере было три вызова, а в другом — два (из‑за условия), то «третий» хук во втором рендере прочитает память «второго» хука — типы и смысл данных собьются, возможны падения и неверное состояние. Поэтому порядок и количество вызовов должны быть стабильными.

**Как соблюдать:** Вызывать все хуки безусловно на верхнем уровне; не вызывать хуки внутри `if`, `for`, вложенных функций. Если логика «иногда нужна» — вызывать хук всегда, а ветвление делать по **результату** хука (по значению, по флагу).

---

## Правило 4: Имена кастомных хуков начинаются с "use"

**Правило:** Функция, которая вызывает хуки и переиспользует логику, должна называться так, чтобы имя **начиналось с "use"** (например, `useAuth`, `useWindowSize`). Так принято называть кастомные хуки.

**Почему:** React не различает «обычную функцию» и «кастомный хук» по сигнатуре — только по месту вызова. Но **линтер** (eslint-plugin-react-hooks) и разработчики по соглашению считают: «если функция начинается с `use` и вызывается из компонента, то это хук» — к ней применяются те же правила (вызывать только на верхнем уровне, только из компонента или другого хука). Если назвать так обычную функцию и вызвать её из цикла или условия, линтер предупредит. Так меньше шанс нарушить правила хуков по невнимательности.

**Итог:** Кастомный хук — это функция с именем вида `useXxx`, которая внутри вызывает хуки; её можно вызывать только из тела компонента или другого хука, на верхнем уровне.

---

## Как React хранит хуки (почему порядок важен)

У каждого компонента React хранит внутренний объект **Fiber**. В нём есть поле для **списка хуков** (связанный список): первый вызов хука при первом рендере создаёт первую запись, второй вызов — вторую и т.д. При следующих рендерах React **не идентифицирует хуки по имени** — он просто идёт по списку по порядку: первый вызов хука в теле функции — первая запись, второй вызов — вторая. Поэтому если между рендерами порядок вызовов изменился (хук пропал или появился в условии), соответствие «вызов — запись» ломается и состояние/эффекты перемешиваются.

**Запомнить:** хуки сопоставляются по **индексу вызова**, а не по имени. Стабильный порядок вызовов — обязательное условие.

## Кратко запомнить

- **Верхний уровень** — не в if/for/вложенных функциях, чтобы порядок вызовов не менялся.
- **Только в компонентах и кастомных хуках** — React привязывает список хуков к компоненту при рендере; в обычных функциях и классах контекста нет.
- **Одинаковый порядок** — React хранит хуки в списке по индексу вызова; сдвиг порядка ломает соответствие и приводит к багам.
- **Имена с "use"** — кастомные хуки так называют по соглашению; линтер проверяет правила для таких функций.
