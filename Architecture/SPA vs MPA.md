# SPA vs MPA

## Краткий ответ для собеседования

**SPA (Single Page Application)** — одно HTML-документ, переходы без перезагрузки страницы; контент подгружается и отрисовывается на клиенте (JS). **MPA (Multi Page Application)** — много отдельных HTML-страниц, переход по ссылке = новый запрос к серверу и новая страница. SPA: быстрая навигация после загрузки, богатый UX, но медленная первая загрузка и сложнее SEO. MPA: быстрая первая отрисовка и проще SEO, но полная перезагрузка при переходах.

---

## Что такое SPA

**Single Page Application** — приложение, в котором браузер загружает один HTML-файл (или минимум разметки), а весь контент и смена «страниц» реализуются на клиенте через JavaScript.

При переходе по «маршруту» не происходит полный запрос нового документа с сервера: подгружаются данные (API), меняется DOM или подставляются компоненты. История и URL часто меняются через History API без перезагрузки.

**Типичный стек:** React, Vue, Angular + роутер; бэкенд отдаёт API (JSON) и один entry-point (index.html + бандл JS).

---

## Что такое MPA

**Multi Page Application** — классическая модель: каждая страница — отдельный запрос к серверу. Сервер отдаёт готовый HTML для этого URL. Переход по ссылке — новый HTTP-запрос, новая загрузка документа, новый рендер.

**Типичный стек:** серверный рендеринг (PHP, Rails, Node с шаблонами, SSR-фреймворки вроде Next в режиме SSR по маршрутам). Каждый маршрут = свой HTML-документ.

---

## Сравнение

| Критерий | SPA | MPA |
|----------|-----|-----|
| **Страницы** | Один документ, «страницы» — состояние в JS | Много документов, у каждого свой URL и HTML |
| **Переходы** | Без перезагрузки, подгрузка данных и смена UI | Полная перезагрузка, новый HTML с сервера |
| **Первая загрузка** | Часто дольше (большой JS, потом данные) | Часто быстрее (готовый HTML) |
| **Навигация после загрузки** | Очень быстрая (уже есть JS и данные) | Каждый переход — новый запрос и рендер |
| **SEO** | Сложнее (изначально пустой или бедный HTML) | Проще (готовый контент в HTML) |
| **Сервер** | Отдаёт статику + API | Генерирует HTML под каждый запрос (или статику по маршрутам) |
| **Состояние** | Живёт в клиенте (память, стейт) | Частично в сессии/куках, частично в каждом документе |

---

## Плюсы SPA

- После первой загрузки навигация быстрая, без мигания и полной перерисовки.
- Удобно делать богатый интерактивный интерфейс (формы, фильтры, дашборды).
- Чёткое разделение: фронт (JS) и бэкенд (API), можно разрабатывать и деплоить отдельно.
- Один раз загруженный код можно переиспользовать между «страницами».

---

## Минусы SPA

- Первая загрузка может быть тяжёлой (большой бандл, потом запросы за данными).
- SEO сложнее: контент появляется после JS; нужен SSR или пререндер, чтобы отдавать полноценный HTML.
- Зависимость от JS: при отключённом или сломанном JS приложение может не работать.
- История, заголовки и мета-теги для «страниц» нужно поддерживать вручную (роутер, SSR/head management).

---

## Плюсы MPA

- Первая отрисовка быстрая: сервер отдаёт готовый HTML с контентом.
- SEO проще: у каждой страницы свой URL и полный HTML.
- Меньше зависимости от тяжёлого клиентского JS; можно постепенно улучшать страницы скриптами.
- Простая модель: одна страница = один ответ сервера.

---

## Минусы MPA

- Каждый переход — полная перезагрузка: мигание, повторная загрузка общих ресурсов (если не кэш).
- Сложнее делать очень «приложенческий» UX (мгновенные обновления, сложные переходы без перезагрузки).
- Больше нагрузки на сервер (генерация HTML для каждого запроса), если не использовать кэш или статику.

---

## Гибридные варианты

- **SSR + гидрация (Next, Nuxt и т.д.):** первая страница — как MPA (готовый HTML), дальше навигация — как SPA (без полной перезагрузки).
- **Статика по маршрутам (SSG):** каждая «страница» — свой HTML-файл, но генерируется при сборке; переходы могут быть SPA-подобными, если подключен клиентский роутер.

---

## Кратко для ответа

SPA — одно приложение в одном документе, маршруты и контент на клиенте, быстрая навигация, но тяжёлый первый загруз и сложнее SEO. MPA — много страниц, каждая со своим HTML с сервера, проще SEO и первая отрисовка, но полная перезагрузка при переходах.
